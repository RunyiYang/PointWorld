# Codex instructions: start RoboTwin two-gripper action training after preprocessing

This document is for Codex after the full RoboTwin/PointWorld training patch has already been unzipped into the PointWorld repo.

Do not unzip or reapply the patch. Do not rerun preprocessing unless the WDS data is missing or invalid. Start from the generated RoboTwin WDS root and train the custom action-space finetuning script.

---

## 0. Expected state before training

Repository root should be the PointWorld `main` branch with the RoboTwin patch already present.

Required patch files:

```bash
test -f tools/robotwin2g/train_robotwin_action.py
test -f tools/robotwin2g/action_model.py
test -f tools/robotwin2g/wds_dataset.py
test -f tools/robotwin2g/smoke_read_wds.py
test -f scripts/robotwin2g_finetune_action_decoder.sh
test -f scripts/robotwin2g_finetune_all.sh
```

Required preprocessed data files:

```bash
WDS_ROOT=/work/runyi_yang/FloWAM/data/robotwin2g_pointworld_wds

test -d "${WDS_ROOT}/train"
test -f "${WDS_ROOT}/action_stats.json"
test -f "${WDS_ROOT}/metadata_rank0.json"
test -f "${WDS_ROOT}/train_sources.txt"
ls "${WDS_ROOT}"/train/*.tar | head
```

For the full dataset, also require test shards:

```bash
ls "${WDS_ROOT}"/test/*.tar | head
```

For tiny one-episode debug data, an empty test split can be valid. For the real full RoboTwin dataset, a missing/empty test split is not acceptable because preprocessing used `TEST_RATIO=0.02` and should hold out some episodes.

## 1. Use a clean raw-data/WDS separation

Do not train from raw RoboTwin HDF5 files. The trainer consumes WebDataset tar shards generated by the preprocessing step.

Correct training input:

```text
${WDS_ROOT}/train/*.tar
${WDS_ROOT}/test/*.tar
${WDS_ROOT}/action_stats.json
${WDS_ROOT}/metadata_rank0.json
```

Incorrect training input:

```text
/work/runyi_yang/FloWAM/data/bartending/episode0.hdf5
/work/runyi_yang/FloWAM/data/dough_rolling/episode0.hdf5
```

The RoboTwin raw data is full-episode HDF5. The preprocessing step converts it to 11-frame PointWorld-style clips with 14-D action labels.

## 3. Make sure the PointWorld checkpoint exists

```bash

```

Check it:


```bash

```

Alternative checkpoints can also be used:

```text

```

For the two-gripper RoboTwin finetuning, prefer `large-droid+behavior/model-best.pt` because it has both DROID and BEHAVIOR pretraining.

## 4. Validate WDS before training

Run a WDS smoke read:

```bash
python tools/robotwin2g/smoke_read_wds.py \
  --wds-root "${WDS_ROOT}" \
  --split train \
  --batch-size 2
```

Expected shapes should look like:

```text

scene_features   (B, 1, N, 33)
robot_flows      (B, 11, Nr, 3)
robot_features   (B, 11, Nr, 14)
action_state     (B, 11, 14)
action_target    (B, 10, 14)
action_mask      (B, 10)
```

Inspect action normalization stats:

```bash
python - <<'PY'
import json
from pathlib import Path
root = Path('/work/runyi_yang/FloWAM/data/robotwin2g_pointworld_wds')
stats = json.loads((root / 'action_stats.json').read_text())
meta = json.loads((root / 'metadata_rank0.json').read_text())
print('action mean dim:', len(stats['mean']))
print('action std dim:', len(stats['std']))
print('metadata:', meta)
assert len(stats['mean']) == 14
assert len(stats['std']) == 14
PY
```

If the smoke read or action stats fail, stop and fix preprocessing before training.

## 5. Run a 10-step training smoke test

Before launching the full run, verify that checkpoint loading, forward pass, loss, backward, and saving work.

```bash
OUT=train_logs/robotwin2g_smoke_action_decoder

python tools/robotwin2g/train_robotwin_action.py \
  --wds-root "${WDS_ROOT}" \
  --pretrained-checkpoint "${PRETRAINED}" \
  --output-dir "${OUT}" \
  --stage action_decoder \
  --batch-size 1 \
  --num-workers 0 \
  --num-epochs 1 \
  --max-steps 10 \
  --eval-every 5 \
  --eval-batches 2 \
  --lr 1e-4 \
  --amp

ls "${OUT}"/checkpoint-last.pt
ls "${OUT}"/checkpoint-best.pt
```

If this fails with CUDA OOM, retry without AMP first only if AMP caused the issue; otherwise lower the scene/robot point count during preprocessing or keep `--batch-size 1`.

If this fails because test shards are absent on a tiny debug sample, the training script defaults to allowing empty test and uses train loss for checkpoint selection. For the full dataset, rerun with `--require-test` during real training.

## 6. Stage 1: train only the new action decoder

Purpose: align the pretrained PointWorld representation to the RoboTwin 14-D two-gripper action output space without changing the PointWorld backbone.

Frozen:

```text
PointWorld backbone
PointWorld scene/robot encoders
DINO/DINOv3 visual modules
```

Trainable:

```text
new action decoder MLP
```

Recommended command:

```bash
WDS_ROOT=/work/runyi_yang/FloWAM/data/robotwin2g_pointworld_wds
PRETRAINED=pretrained_checkpoints/large-droid+behavior/model-best.pt
OUT_STAGE1=train_logs/robotwin2g_action_decoder

BATCH_SIZE=8 \
NUM_WORKERS=4 \
NUM_EPOCHS=20 \
LR=1e-4 \
EVAL_EVERY=500 \
EVAL_BATCHES=20 \
bash scripts/robotwin2g_finetune_action_decoder.sh \
  "${WDS_ROOT}" \
  "${PRETRAINED}" \
  "${OUT_STAGE1}"
```

Direct equivalent:

```bash
python tools/robotwin2g/train_robotwin_action.py \
  --wds-root "${WDS_ROOT}" \
  --pretrained-checkpoint "${PRETRAINED}" \
  --output-dir "${OUT_STAGE1}" \
  --stage action_decoder \
  --batch-size 8 \
  --num-workers 4 \
  --num-epochs 20 \
  --lr 1e-4 \
  --eval-every 500 \
  --eval-batches 20 \
  --require-test \
  --amp
```

Expected outputs:

```text
${OUT_STAGE1}/checkpoint-last.pt
${OUT_STAGE1}/checkpoint-best.pt
${OUT_STAGE1}/checkpoint-step*.pt
```

Monitor logs for:

```text

[eval] step=... mse=... mae=...
```

The best checkpoint is selected by action MAE on the test split.

## 7. Stage 2: finetune PointWorld non-DINO modules plus action decoder

Purpose: adapt the PointWorld 3D/robot representation to the RoboTwin domain after the action decoder is initialized.

Frozen by default:

```text
DINO/DINOv3 visual backbone
```

Trainable:

```text
action decoder
PointWorld non-DINO modules
```

Recommended command:

```bash

PRETRAINED=pretrained_checkpoints/large-droid+behavior/model-best.pt

NUM_WORKERS=4 \
NUM_EPOCHS=20 \

  "${WDS_ROOT}" \

  "${OUT_STAGE2}" \
  "${PRETRAINED}"
```

Direct equivalent:

```bash

  --wds-root "${WDS_ROOT}" \

  --resume "${RESUME}" \

  --batch-size 4 \
  --num-workers 4 \
  --num-epochs 20 \
  --lr 1e-4 \
  --world-lr 2e-5 \
  --eval-every 500 \
  --eval-batches 20 \
  --require-test \
  --amp
```

Expected outputs:

```text
${OUT_STAGE2}/checkpoint-last.pt
${OUT_STAGE2}/checkpoint-best.pt
${OUT_STAGE2}/checkpoint-step*.pt
```

Monitor logs for:

```text

[eval] step=... mse=... mae=...
```

Keep DINO frozen at first. Only add `--unfreeze-dinov3` if there is enough data/GPU memory and the stage-2 validation MAE plateaus due to visual-domain mismatch.

## 9. GPU and batch-size guidance

Start conservative and increase batch size only after the smoke test succeeds.

Suggested settings:

| GPU memory | Stage 1 batch | Stage 2 batch | Notes |
| --- | ---: | ---: | --- |
| 16 GB | 1-2 | 1 | keep workers low |
| 24 GB | 4 | 2 | likely stable |
| 40+ GB | 8 | 4 | default recommendation |
| 80 GB | 8-16 | 4-8 | increase only after monitoring |

If OOM occurs:

1. Lower `BATCH_SIZE`.
2. Lower `NUM_WORKERS` to 0 or 2.
3. Make sure `--amp` is enabled.
4. If still OOM, reduce `MAX_SCENE_POINTS` and `MAX_ROBOT_POINTS` in preprocessing and regenerate WDS.

Do not use `torchrun` for this custom RoboTwin action trainer unless distributed support has been added. The current script is intended as a single-process training entry point.

## 10. What the model is actually learning

Input per WDS clip:

```text
scene_flows / scene features       PointWorld-style scene tensors
robot_flows / robot features       robot/gripper tensors
action_state                       (11, 14)
RGB-D camera fields                initial camera observations
```

Target per WDS clip:

```text
action_target                      (10, 14)
action_mask                        (10,)
```

The loss is action-only:

```text
SmoothL1(predicted normalized action, normalized action_target)
```

The patch intentionally does not train scene-flow loss on RoboTwin observed point clouds. The raw RoboTwin point clouds are per-frame observations and do not guarantee that point index `n` is the same physical point across frames.
